# keel — design for the PGN database refactor

Status: **draft for discussion** — no code exists yet.

`keel` is the tool that will own the CANboat PGN database: a set of YAML
source files that replace `analyzer/pgn.h`, `analyzer/lookup.h` and
`analyzer/fieldtype.h` as the place where NMEA 2000 message definitions are
authored. Like the keel of a ship, everything else is built on top of it:
the C analyzer, `canboat.xml`, `canboat.json`, the docs site and every
downstream consumer.

## 1. Why

Today the database *is* C source. `pgn.h` (10.6k lines) holds a
`Pgn pgnList[]` written in a macro DSL; `lookup.h` (4.7k lines) is an
X-macro file included six times with different macro definitions;
`fieldtype.h` holds the type system. `analyzer-explain` is compiled with
`-DEXPLAIN` (which swaps a union member from val→string functions to
enumerator callbacks) and emits `canboat.xml` through hand-rolled `printf`
calls. XSLT then derives `canboat.json` and `canboat.html`.

Problems, in decreasing order of pain:

1. **The rules are invisible.** The XSD describes the *shape* of the XML,
   but the real restrictions on a PGN live scattered across
   `pgn.c:checkPgnList()`, `analyzer-explain.c:getMinimalPgnLength()`,
   `fieldtype.c:fillFieldType()` and pure convention. A contributor finds
   out about them one `logAbort` at a time, after writing C.
2. **Contributing requires writing C** in a macro dialect with puns like
   `unit = "=275"` meaning "match field" and `ID_AND_NAME` pinning a frozen
   identifier that is otherwise silently derived from the display name.
3. **One 10.6k-line file** means every pair of concurrent PRs conflicts.
4. **The XML emission is implicit.** Presentation decisions (sentinel
   emission conditions, `%g` formatting, bit-offset suppression after
   variable fields, camelization quirks) are interleaved with the data walk
   in `analyzer-explain.c` and cannot be tested in isolation.

## 2. The decision

* The database moves to **YAML files, one file per `pgnList` entry** (i.e.
  per PGN *variant*) plus one file per lookup enumeration. YAML was chosen
  over per-PGN XML fragments because the source is read (in PR review,
  `git blame`, archaeology) far more often than it is written, and YAML is
  simply much cleaner to read. The cost — needing an XML emitter that
  reproduces today's output — is paid once and verified by a byte-for-byte
  golden test at switchover.
* Writing complete new YAML by hand is *not* the intended workflow: a
  **local web editor** (keel-served backend + no-build vanilla JS frontend)
  guides contributors and applies every rule before a PR is created. The
  same rule engine runs in the editor, in `keel check`, and in CI —
  if the editor is green, CI is green.
* `pgn.h`, `lookup.h` **and `fieldtype.h` are generated** from the YAML and
  committed, so building the C analyzer never requires the keel toolchain.
* `canboat.xml` is generated by `keel`, **byte-identical at switchover**.
  The existing XSLT chain (`canboat2json.xslt` → `canboat.json`,
  `canboat.xsl` → `canboat.html`) is kept unchanged.
* The **v1 output dies**: `-v1`, `getV1Type()`, the `v1Type` member and
  `pgns2json.xslt` are removed, not ported.
* The `-explain` **text format moves to `keel explain`**. After that,
  `analyzer-explain.c` and the whole `-DEXPLAIN` dual-compile mechanism are
  deleted.

## 3. Source tree

```
database/
  pgns/                      one file per pgnList entry, flat for now
    059392-isoAcknowledgement.yaml
    130306-windData.yaml
    130845-simnetKeyValue.yaml
    0xE800-fallback-....yaml         (fallback/catch-all entries are files too)
    ...                              (Actisense/iKonvert BEM pseudo-PGNs too;
                                      they are needed at analyzer runtime)
  j1939/
    pgns/...                         J1939 variant set (today: pgn-j1939.h)
  lookups/
    WIND_REFERENCE.yaml              one enumeration per file
    MANUFACTURER_CODE.yaml
    ...                              (pair, triplet, bit and fieldtype
                                      lookups all live here)
  fieldtypes.yaml                    the FieldType hierarchy
  physicalquantities.yaml            the PhysicalQuantity list
```

Naming: `<pgn, zero-padded>-<id>.yaml`, flat. If a PGN family grows enough
variants to be unwieldy we can flip to `pgns/<pgn>/<id>.yaml` later; the
loader must not care.

### 3.1 Authored data only — never consequences

The single most important format rule: **a source file contains only
decisions**. Everything derivable is computed by `keel` at generation time,
exactly as the C computes it today, and appears only in the generated
artifacts:

| Derived, never in YAML | Computed from |
|---|---|
| `Order` | field position |
| `BitOffset` / `BitStart` | sum of preceding field sizes |
| `FieldCount`, `Length` / `MinLength` | the field list |
| `RangeMin` / `RangeMax` | bits, sign, resolution, offset |
| `UnknownValue` / `OutOfRangeValue` / `ReservedValue` | bits, sentinel rules, lookups |
| camelCase anything | nothing — ids are explicit, see below |

Reviewers still see the computed fallout, because the generated
`docs/canboat.xml` / `docs/canboat.json` stay committed and appear in the
same PR diff.

One deliberate exception: **`id` is always explicit**, on PGNs and on
fields. Ids are the frozen downstream contract (`tools/contract.py`, the
upward-compatibility workflow, ts-pgns/canboatjs identifiers). Deriving
them from display names — with `ID_AND_NAME` / `.camelDescription` pinning
as an escape hatch — is exactly the kind of magic this refactor removes.
With explicit ids, renaming a `name` can never silently change an `id`,
and the `Reserved`/`Spare` numbered-suffix convention becomes a validator
rule instead of runtime C.

### 3.2 PGN file example

```yaml
pgn: 130306
id: windData
description: Wind Data
type: single            # single | fast | isotp
complete: true          # or missing: [Fields, Interval, SampleData, ...]
priority: 2
interval: 100           # ms; absent = unknown, on-demand: irregular: true
explanation: >-
  Direction and speed of Wind. Direction is relative to the vessel or
  true/magnetic, indicated by the Reference field.
fields:
  - id: sid
    name: SID
    type: UINT8
  - id: windSpeed
    name: Wind Speed
    type: UFIX16
    unit: m/s
    resolution: 0.01
  - id: windAngle
    name: Wind Angle
    type: UFIX16
    unit: rad
    resolution: 0.0001
  - id: reference
    name: Reference
    bits: 3
    lookup: WIND_REFERENCE
  - reserved: 21
```

A proprietary variant, showing first-class keys where the C used puns:

```yaml
pgn: 130845
id: simnetKeyValue
description: "Simnet: Key Value"
type: fast
fields:
  - id: manufacturerCode
    name: Manufacturer Code
    bits: 11
    lookup: MANUFACTURER_CODE
    match: 1857            # was: unit = "=1857" in C
  - reserved: 2
  - id: industryCode
    name: Industry Code
    bits: 3
    lookup: INDUSTRY_CODE
    match: 4
  # ...
repeating1:
  count: 3                 # fields in the repeating set
  start: 6                 # 1-based order of the first repeating field
  countField: 5            # absent = repeat until data exhausted
```

Other field-level keys, replacing today's macro zoo: `signed`, `offset`
(excess-K), `precision`, `primaryKey: true`, `proprietary: true` (field only
present in proprietary range), `specialValues: N` (the `SPECIAL_VALUES`
sentinel override), `dynamicFieldLength: {overhead: N}`,
`lookupIndirect: {name: ..., order: N}`, `lookupBits`, `lookupFieldtype`,
`description`, `condition`. PGN-level: `fallback: true`, `url`,
`researchDoc`, `notes`, `repeating2`.

The research commentary currently living as block comments in `pgn.h` is
ported into `notes:` keys (load-bearing prose survives round-tripping
through tools) or `#` comments (asides).

### 3.3 Lookup file example

```yaml
name: WIND_REFERENCE
bits: 3
values:
  0: True (ground referenced to North)
  1: Magnetic (ground referenced to Magnetic North)
  2: Apparent
  3: True (boat referenced)
  4: True (water referenced)
```

Triplet (indirect), bit, and fieldtype lookups use `triplets:`, `bitValues:`
and `fieldtypeValues:` maps in the same shape.

### 3.4 fieldtypes.yaml

Carries the full `FieldType` hierarchy (name, base type, bits, sign,
resolution, unit, physical quantity, sentinel convention, URLs and prose).
The print functions remain hand-written C; the YAML names them
symbolically (`print: NUMBER` → `.pf = fieldPrintNumber`) via a fixed map
in the generator.

### 3.5 YAML discipline

* Parsed with a **strict safe loader, YAML core schema** (no Norway
  problem: `no` is a string only if quoted — the canonicalizer quotes it).
* `keel fmt` machine-rewrites every file into one canonical style: key
  order fixed, strings quoted where ambiguous, folded scalars for prose.
  Files are machine-written by the editor anyway; `fmt` keeps hand edits
  from drifting.

## 4. The keel tool

**Rust** (edition 2024), a single crate in `keel/` producing one static
binary — no interpreter, no venv, near-zero dependencies (`yaml-rust2` for
parsing; floats print via libc `snprintf` declared extern; the editor will
use a minimal HTTP crate, not an async framework). Launched through thin
shims committed next to it: `keel/keel` (sh) and `keel/keel.cmd` (Windows),
which run the prebuilt binary, else `cargo build --release`, else download
a release binary from GitHub.

Why Rust (revisited from an earlier Python draft): a static binary is the
cleanest contributor experience on every platform; the rule engine compiles
to **wasm** for the phase-2 hosted editor (strictly better than Pyodide);
and the model/derive crates can eventually be shared with canboat-rs, so
the YAML database feeds a Rust decoder without the canboat.json detour.
Range arithmetic uses i128 before the final f64 conversion, which is what
makes the derived ranges bit-exact against the C.

One deliberate exception: the **bootstrap converter**
(`canboat.xml` + `fieldtype.h` → `database/`) stays a small Python package
in `keel/bootstrap/` (PyYAML, own venv, `keel-bootstrap` entry point). It
is a one-shot migration tool that dies at switchover; until then it also
re-syncs `database/` when master's `pgn.h` changes, and the Rust emitter
agreeing byte-for-byte with the Python-written tree is itself a strong
cross-language check.

```
keel check       load + validate everything, print rule violations, exit≠0 on any
keel generate    check, then write ALL generated artifacts (see §6)
keel generate --check   regenerate to a temp dir, diff against committed
                 artifacts, exit≠0 on drift (never writes; for hooks/CI)
keel fmt         canonicalize YAML in place
keel new         scaffold a new PGN file (e.g. keel new 130846 --manufacturer=Furuno)
keel edit        start the local web editor (serves SPA + validation API)
keel explain     the old analyzer-explain text output, from YAML
keel diff        contract-level diff of two database states (wraps tools/contract.py)
```

Internal architecture — one load path for everything:

```
loader (strict YAML) ─► model (plain dataclasses)
                           │
                 rule engine (all invariants, §5)
                           │
        ┌──────────┬───────┴────────┬───────────────┐
   xml emitter   c emitter    text emitter    editor API
   canboat.xml   pgn.h        keel explain    /validate /save /model
   (+ngt/ik)     lookup.h
                 fieldtype.h
```

The rule engine is the heart: **every invariant is a named rule with an ID,
a message template, and a source location** (file + YAML path), so the
editor, `keel check` and CI all report identically, e.g.:

```
R03 pgns/130306-windData.yaml fields: bit sizes sum to 52; a PGN must fill
    whole bytes (add reserved bits: 4 missing)
```

## 5. The rules (initial inventory)

Ported from `pgn.c`, `fieldtype.c`, `analyzer-explain.c` and convention.
Numbering is illustrative; the set will grow.

Shape (JSON Schema over the YAML):
* R01 required keys, allowed keys, types, enum values per object.

Intra-PGN semantics:
* R02 PGN number lies in a valid `pgnRange`; PDU1 PGNs end in 0x00.
* R03 field bit sizes up to the first repeating field sum to whole bytes
      (enforced as a hard error in derive - every length depends on it).
* R04 fixed single-frame PGNs are exactly 8 bytes (59904 excepted);
      variable single-frame ≤ 8 bytes; packet `type` agrees with the
      range's type (ISO-TP / mixed exceptions).
* R05 repeating sets are self-consistent (start/count in range,
      `countField` references an integer field before the set, or is
      absent for repeat-until-exhausted).
* R06 `dynamicFieldLength` precedes its `DYNAMIC_FIELD_VALUE` field;
      overhead sane.
* R07 proprietary-range PGNs start with Manufacturer Code / reserved /
      Industry Code, or carry `missing: [MissingCompanyFields]`.
* R08 lookup references resolve with the right kind; a named value that
      cannot fit the field is an error (dead table entry; shared *bit*
      enumerations excepted); a mere width disagreement with the declared
      lookup width is a warning (inherited pgn.h laxity, see FINDINGS.md).
* R09 sentinel logic: `specialValues` override in range; lookups naming
      values in the sentinel region reduce the reserved count; no
      sentinels on match fields or 64-bit fields.
* R10 fieldtype constraints: resolution/offset/unit may not contradict the
      base type; unit implies a known physical quantity (hard error in
      derive - it poisons every resolved attribute).
* R11 max 33 fields (raiseable, but enforced to match the C runtime).
* R12 `reserved`/`spare` id suffix convention.

Cross-file:
* R20 (pgn, match-set) combinations are unique (a duplicate makes the
      later variant unreachable at runtime); at most one match-less
      catch-all per PGN number. Temporarily a warning: two inherited
      duplicates exist (FINDINGS.md F1); re-harden once resolved.
* R21 ids unique per scope (PGN ids globally; field ids within a PGN).
* R22 every lookup file is referenced (warning), every reference resolves
      (error).
* R23 fieldtype hierarchy is a DAG rooted in types with print functions.

Cross-release (CI only):
* R30 `tools/contract.py` classification against the previous release —
      frozen ids, wire-encoding changes, etc. Unchanged from today.

Sort order of the output is *imposed* by the generator (PGN ascending,
fallback first), so today's "list must be sorted" invariant disappears
rather than being checked.

## 6. Generated artifacts

All committed, all under `keel generate`, CI fails if regeneration changes
anything (same policy as `docs/canboat.json` today).

| Artifact | Notes |
|---|---|
| `docs/canboat.xml` | **Byte-identical to today's output at switchover.** Quirk-compatible: C `%g` float formatting (needs a small `%g` mimic — Python formats floats differently), the `RangeMax` UINT64_MAX special case, entity escaping, the 4-space `<Priority>` indent bug, element order. The full quirk inventory with keep/cleanup/fix recommendations lives in **[QUIRKS.md](./QUIRKS.md)**. After switchover the golden file retires; `contract.py` governs change. |
| `docs/canboat.json`, `.html` | unchanged XSLT chain, still driven from the XML. |
| `analyzer/pgn.h` | generated `Pgn pgnList[]`. May keep today's initializer shape initially; can later simplify (the `unit="=N"` pun can stay *internal* to the generated C or be replaced together with the runtime code that reads it). |
| `analyzer/lookup.h` | generated; the six-way X-macro include collapses — only the analyzer's val→string direction is needed once `-DEXPLAIN` dies. Plain static tables are an option. |
| `analyzer/fieldtype.h` | generated from `fieldtypes.yaml`; print functions stay hand-written in `fieldtype.c`. |
| `analyzer/pgn-j1939.h` | same generator, `database/j1939/` tree. |
| Actisense/iKonvert XML | today's `-explain-ngt-xml` / `-explain-ik-xml`, now `keel generate --actisense/--ikonvert`; the BEM pseudo-PGN YAML also feeds `pgn.h` (runtime decoding needs it). |
| `keel explain` text | replaces `analyzer-explain -explain`; on-demand, not committed. |

Deleted when the migration completes: `analyzer-explain.c`, the `-DEXPLAIN`
build variants, `LOOKUP_*_MEMBER` union swapping, all v1 machinery
(`pgns2json.xslt`, `getV1Type`, `v1Type`), `ID_AND_NAME` /
`camelDescription` pinning, `camelCase()` / `camelize()`, and most of the
macro DSL in `pgn.h`.

Version stamping (`fixup-version.py`, `SCHEMA_VERSION`) moves into
`keel generate`, reading the same source of version truth as the C build.

## 7. The editor

**Local web only.** `keel edit` starts a small local HTTP server (a
minimal Rust HTTP crate, no async framework) in the clone and opens the
browser:

* Backend endpoints: `GET /model` (the loaded database), `POST /validate`
  (run the rule engine on a candidate document, return rule violations
  with YAML paths), `POST /save` (canonicalize + write + full check),
  `GET /lookups`, `GET /fieldtypes` (populate pickers).
* Frontend: a single HTML file, vanilla JS, no build step — contributors to
  a C project should not need npm. Served straight from `keel/web/`.
* Centerpiece: a **live bit-layout grid** — the frame drawn byte by byte,
  fields packing left to right as you add them, byte boundaries visible,
  misalignment red (with "add N reserved bits" one click away), sentinel
  regions shaded, repeating sets bracketed. This is the guidance a
  starting developer actually needs.
* Pickers over free text everywhere sensible: fieldtypes, units/physical
  quantities, existing lookup enumerations (searchable — steering people
  to reuse `YES_NO` instead of minting `ENABLED_DISABLED`).
* Templates: proprietary-PGN scaffold with the company preamble
  pre-filled; fast/single defaults by PGN range.
* Save is gated: a file that fails any rule cannot be written (the CLI can
  `--force` for work-in-progress; the editor does not).

Because the editor calls the same rule engine as `keel check` and CI, its
green state *is* PR readiness.

## 8. Contribution workflows

* **Developers**: clone → `keel edit` (or hand-edit YAML) → `keel check` →
  `keel generate` → commit source + regenerated artifacts → PR. CI re-runs
  the same two commands and `contract.py`.
* **Non-developers**: GitHub issue forms with structured fields for the
  simple cases (new lookup value, new simple PGN). An Action parses the
  submission, runs the rule engine, and either opens a draft PR with the
  generated YAML or reports the rule violations back as an issue comment.
* **Later, if wanted**: the same SPA hosted statically on the docs site
  with the validator compiled to the browser (the rule-engine crate
  built for wasm) — author a
  definition with zero setup, download the YAML, attach it to an issue.
  Nothing in the architecture blocks this; nothing depends on it.

### 8.1 Local enforcement: git hooks

A committed **`.githooks/pre-push`** hook verifies, before anything leaves
the machine, that the push is rule-clean and that the committed generated
artifacts are current:

```
keel check
keel generate --check     # regenerate + diff, fail on drift
```

Both run the prebuilt keel binary (no C compile), so the hook costs
well under a second; it skips
itself when the pushed range touches neither `database/` nor `keel/`.

Git deliberately does not clone hooks, so activation goes through
`core.hooksPath`: the first time `keel` runs in a clone it sets
`git config core.hooksPath .githooks` itself (with a one-line notice);
`make setup` covers anyone who never runs keel.

Hooks are advisory — `--no-verify` bypasses them and github.com offers no
custom server-side pre-receive hooks — so **CI remains the authoritative
gate**: the same two commands run as a required status check under branch
protection. The hook exists to catch drift in seconds locally instead of
minutes later in CI.

## 9. Migration plan

Each step lands separately, each with a hard gate. The old and new paths
coexist until step 5.

1. **Converter (bootstrap).** Parse today's `docs/canboat.xml`, strip all
   derived data (§3.1), emit the YAML tree.
   *Gate:* `keel generate` output byte-diffs empty against today's
   `canboat.xml`. The strip-and-recompute round trip self-tests the
   derivation logic across all ~800 entries — every `%g` case, every
   sentinel rule, every quirk.
2. **Rule engine.** Port every invariant from `pgn.c` / `fieldtype.c` /
   `getMinimalPgnLength()` into named rules.
   *Gate:* the current database passes `keel check` with zero violations;
   deliberately broken fixtures trip each rule.
3. **C generator.** Emit `pgn.h`, `lookup.h`, `fieldtype.h`, `pgn-j1939.h`.
   *Gate:* the generated headers compile; the *old* XML path
   (`analyzer-explain -explain-xml`) run against the generated headers
   still reproduces `canboat.xml`; the analyzer test suite passes.
4. **Comment porting.** Move the `pgn.h` research prose into `notes:` keys.
   Hand work, reviewable in slices.
5. **Switchover.** `make generated` calls `keel generate`; delete
   `analyzer-explain.c`, `-DEXPLAIN`, v1, the macro DSL, the C-side checks
   now owned by rules (keep only cheap runtime assertions on table
   consistency). Trim `AGENTS.md`/`CONTRIBUTING.md` accordingly.
6. **Editor.** `keel edit`, then the issue-form bot. Deliberately last —
   it needs a stable data layer, and steps 1–5 already deliver most of the
   value (no more merge conflicts, explicit rules, no C required to
   contribute).

## 10. Open questions

* Exact YAML key vocabulary (§3.2 is a proposal, to be finalized against
  the full field-macro inventory during step 1).
* Whether generated `pgn.h` keeps the current initializer shape forever or
  gets simplified together with runtime code (`unit="=N"` pun removal,
  static lookup tables) in a follow-up.
* Whether `database/` is the right name for the source tree (`definitions/`
  and `db/` are the alternatives considered).
* How `keel` versioning interacts with release-please and the
  `fixup-version.py` stamping (mechanical, decide during step 5).
